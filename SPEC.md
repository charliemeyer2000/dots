# `charliemeyer2000/dots` â€” Project Spec

## Overview

A fully declarative, portable development environment using Nix + nix-darwin + home-manager. The goal is: clone repo, run one command, enter 1Password password, and have a complete working environment on any machine â€” Mac or Linux, personal or work, cloud VM or HPC node.

The repo is **public**. Secrets are never committed â€” only templates that reference 1Password. The actual secret values are injected at activation time via the `op` CLI.

## Module Specs

### `modules/base.nix`

Installed on every host regardless of OS. Contains:

**Packages (via nixpkgs):**
- Shell tools: `git`, `ripgrep`, `fd`, `fzf`, `jq`, `curl`, `wget`, `htop`, `tree`, `tmux`
- Dev tools: `gh` (GitHub CLI), `direnv`, `starship`

- Package managers: `pnpm`, `uv`
- Cloud/infra: `awscli2`, `_1password-cli`
- Fonts: managed here or in `home/fonts.nix`

**Nix settings:**
- Enable flakes + nix-command
- Pin nixpkgs input (never use unstable without pinning)

**Shell:**
- Set default shell to zsh

### `modules/darwin.nix`

Mac-specific system config. Contains:

**System defaults (declarative):**
```nix

  dock.autohide = true;

  NSGlobalDomain.InitialKeyRepeatDelay = 15;
```

**Security:**
```nix

```

**Services:**
```nix

```

Tailscale runs as a launchd daemon, starts on boot, no manual intervention needed. Run `tailscale up` once after first activation to authenticate.

**Homebrew (via nix-homebrew module):**
- Enable homebrew management
- Set `onActivation.autoUpdate = true`
- Set `onActivation.cleanup = "zap"` â€” declared list is source of truth

---

### `modules/secrets.nix`

Sets up the 1Password-based secrets flow. Contains:

**Assumptions:** `_1password-cli` is already installed (it's in base.nix). User has run `op signin` before `darwin-rebuild switch`.

**Activation script** that runs on every `darwin-rebuild switch`:
```bash
# Only runs if op session is active
if op whoami &>/dev/null; then
  op inject -i ~/.dots/secrets/secrets.zsh.tmpl -o ~/.env.local
  op inject -i ~/.dots/secrets/aws.tmpl -o ~/.aws/credentials
fi
```

**Note:** If op session is not active, secrets injection is skipped silently. User must run `op signin && darwin-rebuild switch` to inject secrets on a fresh machine.

## Secrets Architecture

### The pattern

Two types of secrets:

**1. Global credentials (injected into PATH via zsh)**

Stored in `secrets/secrets.zsh.tmpl` (committed):
```bash
export GITHUB_TOKEN="{{ op://Personal/GitHub/token }}"
export ANTHROPIC_API_KEY="{{ op://Personal/Anthropic/api-key }}"
export OPENAI_API_KEY="{{ op://Personal/OpenAI/api-key }}"
export AWS_ACCESS_KEY_ID="{{ op://Personal/AWS/access-key-id }}"
export AWS_SECRET_ACCESS_KEY="{{ op://Personal/AWS/secret-access-key }}"
```

Injected to `~/.env.local` (gitignored) via `op inject`. Sourced in zshrc on every shell start.

**2. Per-project overrides (via direnv)**

Each project repo can have an `.envrc` (committed to that repo, not dots):
```bash
# .envrc in a project
export AWS_PROFILE=dueflow-prod
export DATABASE_URL="$(op read 'op://Dueflow/DB/url')"
```

direnv runs this automatically on `cd`. Unsets on `cd` out. No manual switching.

### Adding a new API key

**Via 1Password UI:** Create item â†’ add credential field â†’ update `secrets.zsh.tmpl` to reference it â†’ run `op inject -i secrets/secrets.zsh.tmpl -o ~/.env.local`

**Via CLI:**
```bash
# store it
op item create --category=login --title="ServiceName" --vault=Personal credential=<paste-key>

# or add to existing item
op item edit "ServiceName" "api-key=<paste-key>"

# then re-inject
op inject -i ~/.dots/secrets/secrets.zsh.tmpl -o ~/.env.local && source ~/.env.local
```

Consider writing a shell function `store-key <vault-item> <field> <value>` that wraps this and re-injects.

## Bootstrap Script (`bootstrap.sh`)

For a brand new machine (EC2, new Mac, HPC node):

```bash
#!/usr/bin/env bash
set -e

# 1. Install Nix (Determinate Systems installer â€” better defaults, flakes enabled)
if ! command -v nix &>/dev/null; then
  curl --proto '=https' --tlsv1.2 -sSf https://install.determinate.systems/nix | sh
  . /nix/var/nix/profiles/default/etc/profile.d/nix-daemon.sh
fi

# 2. Clone dots (public repo, no auth needed)
if [ ! -d "$HOME/.dots" ]; then
  git clone https://github.com/charliemeyer2000/dots ~/.dots
fi

# 3. Detect host
if [[ "$(uname)" == "Darwin" ]]; then
  HOST="darwin-minimal"
else
  HOST="linux-ec2"
fi

# 4. Apply config (installs 1password-cli as part of base)
nix run nix-darwin -- switch --flake ~/.dots#$HOST

# 5. Sign into 1Password and inject secrets
echo "Sign into 1Password to inject secrets..."
eval $(op signin)
op inject -i ~/.dots/secrets/secrets.zsh.tmpl -o ~/.env.local
op inject -i ~/.dots/secrets/aws.tmpl -o ~/.aws/credentials

echo "Done. Open a new terminal."
```

**Full Mac setup** (personal machine) would pass `--host darwin-personal` flag to also install GUI apps.

## Things That Cannot Be Nixified (MANUAL.md candidates)

- **ROS2** â€” requires SIP disabled, installs to `/opt/ros`. Manual install, document the steps.
- **Any kext-based installs** â€” hardware drivers, some VPN clients
- **Mac App Store apps** â€” no programmatic install. List them in MANUAL.md so you remember to install on a new Mac.
- **App-internal configs** â€” Raycast extensions/keybindings, Cursor settings (use Settings Sync), browser profiles

## Repo Tooling & CI

### Flake structure â€” `flake-parts`

Use `flake-parts` to keep `flake.nix` from becoming a monolith as hosts, CI jobs, and dev shells accumulate. It provides a module system for flake outputs â€” each concern lives in its own file rather than all being inlined.

```
flake.nix           # thin entry point, imports parts
parts/
  hosts.nix         # darwinConfigurations + nixosConfigurations
  checks.nix        # nix flake check targets
  devshell.nix      # nix develop environment
  formatter.nix     # nix fmt config
```

### Formatting â€” `alejandra`

Declare the formatter in `flake.nix`:
```nix
formatter.aarch64-darwin = pkgs.alejandra;
formatter.x86_64-linux = pkgs.alejandra;
```

Then `nix fmt` formats all `.nix` files consistently. One formatter, no config needed.

### Linting â€” `deadnix` + `statix`

- **deadnix** â€” finds unused variables and imports. Catches cruft as config grows.
- **statix** â€” warns on common Nix antipatterns. Useful while learning Nix idioms.

Both run in CI and as pre-commit hooks.

### Git Hooks â€” `pre-commit-hooks.nix`

Declared in the flake, installed automatically when entering the dev shell. Anyone who checks out dots and runs `nix develop` gets the hooks â€” no manual `pre-commit install` step.

```nix
# parts/checks.nix
pre-commit-hooks.nix.hooks = {
  alejandra.enable = true;   # nix formatting
  deadnix.enable = true;     # unused nix code
  statix.enable = true;      # nix antipatterns
  shellcheck.enable = true;  # bootstrap.sh, audit.sh
};
```

### Dev Shell

A `devShell` for working on dots itself. Enter with `nix develop`:

```nix
devShells.default = pkgs.mkShell {
  packages = with pkgs; [
    alejandra
    deadnix
    statix
    nil          # nix LSP for editor integration
  ];
  shellHook = ''
    ${config.pre-commit.installationScript}
  '';
};
```

### Task Runner â€” `justfile`

A `justfile` at repo root so you don't have to remember full command incantations:

```makefile
# justfile
default:
  just --list

# rebuild current machine
switch:
  nix flake check && darwin-rebuild switch --flake ~/.dots#darwin-personal

# rebuild and show diff of what changed
switch-dry:
  darwin-rebuild build --flake ~/.dots#darwin-personal && nvd diff /run/current-system result

# update all flake inputs (do this intentionally, not automatically)
update:
  nix flake update && git add flake.lock && git commit -m "chore: update flake inputs"

# format all nix files
fmt:
  nix fmt

# run all checks
check:
  nix flake check

# enter dev shell
dev:
  nix develop
```

Run with `just switch`, `just update`, etc. Install `just` via base.nix.

### CI â€” GitHub Actions

Two workflows:

**`ci.yml` â€” runs on every push/PR:**
```yaml
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: DeterminateSystems/nix-installer-action@main
      - uses: DeterminateSystems/magic-nix-cache-action@main  # uses Cachix under the hood
      - run: nix flake check
      - run: nix fmt -- --check   # fails if formatting is wrong
```

**`build-darwin.yml` â€” runs on push to main (macOS runner):**
```yaml
jobs:
  build:
    runs-on: macos-latest
    steps:
      - uses: DeterminateSystems/nix-installer-action@main
      - uses: cachix/cachix-action@v14
        with:
          name: charliemeyer-dots
          authToken: ${{ secrets.CACHIX_AUTH_TOKEN }}
      - run: nix build .#darwinConfigurations.darwin-personal.system
```

The macOS build pushes results to Cachix so new machines pull pre-built derivations instead of compiling from source. Cuts bootstrap time significantly.

### Cachix â€” Binary Cache

Sign up at cachix.org, create a cache called `charliemeyer-dots`. Free tier is sufficient for personal use.

Add to `flake.nix`:
```nix
nixConfig = {
  extra-substituters = [ "https://charliemeyer-dots.cachix.org" ];
  extra-trusted-public-keys = [ "charliemeyer-dots.cachix.org-1:<your-public-key>" ];
};
```

New machines automatically pull from your cache before falling back to building. The CI workflow above populates the cache on every main branch build.

### nixpkgs Input Updates

Never update nixpkgs accidentally. The pattern:
- Pin `nixpkgs` to a specific commit in `flake.lock`
- Update intentionally via `just update` â€” this bumps all inputs and commits the lockfile in one step
- Review the commit diff before running `just switch` to understand what changed
- If something breaks after an update, `git revert` the lockfile commit and rebuild

## Key Design Decisions

**Why public repo?** Templates are committed, not secrets. `op inject` produces the real values locally. Safe to be public.

**Why 1Password over agenix/sops?** You already use 1P. The `op` CLI is straightforward. agenix adds complexity without adding much for a solo developer setup.

**Why Determinate Systems installer over official?** Flakes enabled by default, better uninstall story, sensible defaults.

**Why pin nixpkgs?** Unpinned = environment drifts over time. Update intentionally with `nix flake update`, not accidentally.

**Why `cleanup = "zap"` for homebrew?** Declared list is source of truth. Prevents cask accumulation. A new Mac gets exactly what's declared, nothing more.

**Why colima over Docker Desktop?** Lighter, more scriptable, can be managed as a nix-darwin service. No GUI needed for most dev use cases.

**Why flake-parts?** Keeps `flake.nix` from becoming a 500-line file. Each concern (hosts, CI, dev shell, formatter) lives in its own `parts/` file. Easier to navigate as config grows.

**Why alejandra over nixfmt?** Both are fine. Alejandra is opinionated and requires zero config â€” just run `nix fmt` and it works.

**Why a justfile?** The full `darwin-rebuild switch --flake ~/.dots#darwin-personal` incantation is annoying to type and easy to get wrong. `just switch` is the single thing everyone on the machine runs. It also documents available operations for anyone (including future-you) reading the repo.

**Why Cachix?** First `nix build` on a new machine without a cache takes 20-40 minutes compiling. With Cachix populated by CI, it's 2-3 minutes of downloads. Free for personal use, trivial to set up.

**Why CHANGELOG.md?** Dots accumulate changes over months. A brief dated log ("2025-06: switched from nvm to nix-managed node because X") means you understand your own config six months later.
